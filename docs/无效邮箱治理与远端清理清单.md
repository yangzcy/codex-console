# 无效邮箱治理与远端清理清单

本文用于给 `cc.snacktruckmall.shop` 对应项目提供一份可执行的中文清单，目标是：

- 只保留有用邮箱
- 降低坏邮箱重复参与任务的概率
- 不拖慢现有队列执行
- 不把“远端同步删除”做成高风险主链路动作

这份清单从“功能稳定性”和“性能收益”两个角度出发，优先考虑不破坏现有注册、延后重试、队列吞吐和配置安全。

## 结论

当前阶段，有必要做的是“无效邮箱治理”，而不是“实时同步删除 Cloudflare 远端数据”。

更准确地说：

- 必须尽快解决的是：坏邮箱不要再次被新任务选中
- 可以逐步建设的是：异步、低优先级的远端清理能力
- 暂时不建议做的是：把远端删除直接塞进注册任务主链路

原因很明确：

- 队列任务当前主要绑定的是邮箱服务配置，不是具体邮箱实体
- 延后任务仍可能继续使用同一上下文
- 实时远端删除会引入额外网络 IO 和失败点
- `cloud_mail` 当前并不具备真正删除远端邮箱的完整闭环

## 必须做

以下事项收益高、风险低，建议优先完成。

### 1. 对无效邮箱做“逻辑淘汰”，不要让它继续进入新任务分配

目的：

- 直接减少坏邮箱复用
- 提高注册成功率
- 降低无意义重试

最小要求：

- 给邮箱打状态标签
- 标记后禁止新任务再次选用
- 保留当前运行任务和延后任务的正常执行

建议状态至少包括：

- `in_use`
- `deferred_hold`
- `suspect`
- `invalid_hard`
- `quarantined`
- `purge_ready`
- `deleted`

### 2. 区分“暂时失败”和“永久无效”

不要把所有失败都当作垃圾邮箱处理。

应当优先隔离但不要直接删除的场景：

- `email_otp_timeout`
- `network_timeout`
- `primaryapi_server_error`
- 上游临时波动

可以更快进入清理候选的场景：

- `mailbox_invalid_domain`
- 邮箱格式或域名明确非法
- 上游明确返回邮箱不可再用

### 3. 将清理动作移出热路径

注册任务线程里只做轻量状态更新，不做远端删除。

理由：

- 减少队列阻塞
- 避免删除接口失败反向拖慢注册
- 避免误删延后重试仍需使用的邮箱

### 4. 对延后任务设置保护条件

只要满足以下任一条件，就不应该清理邮箱：

- 存在 `running` 任务正在使用
- 存在 `deferred` 任务等待继续执行
- 最近刚创建或刚使用
- 已经形成成功账号记录
- 已被标记为“该邮箱已注册过 OpenAI”的历史占位记录

## 建议做

以下事项不是必须立刻上线，但属于中期最有价值的治理动作。

### 1. 新增邮箱生命周期表

建议单独新增一张邮箱治理表，例如 `mailbox_inventory`，不要直接复用 `accounts` 作为清理依据。

建议字段：

- `mailbox_email`
- `service_type`
- `service_id`
- `state`
- `last_reason_code`
- `fail_count`
- `last_task_uuid`
- `last_used_at`
- `quarantined_at`
- `purge_after`
- `remote_delete_status`
- `remote_delete_attempts`
- `remote_deleted_at`

这样做的好处：

- 任务执行和邮箱治理彻底解耦
- 可以先记录、后观察、再决定是否删除
- 后续更容易做批处理、审计和回滚

### 2. 做一个异步清理器

建议用定时任务或后台 worker 做，不占用注册主线程。

推荐策略：

- 每次只处理小批量
- 只处理 `purge_ready` 状态的邮箱
- 删除前再次校验是否仍被运行中或延后任务引用
- 删除失败时退避重试

建议参数：

- 单批处理量：`50` 到 `200`
- 删除并发：`2` 到 `5`
- 运行时段：优先低峰期

### 3. 先做“隔离观察”，再做“物理删除”

更稳的两阶段流程：

1. 先把邮箱打入 `quarantined`
2. 观察一段时间，确认没有任务再用
3. 再推进到 `purge_ready`
4. 最后由清理器做物理删除

这一步能显著降低误删风险。

### 4. 先支持可观测性，再追求彻底清理

建议先补齐这些观测项：

- 每日新增邮箱数
- 每日进入 `invalid_hard` 数
- 每日进入 `quarantined` 数
- 每日实际远端删除成功数
- 每类 `reason_code` 的失败占比
- 坏邮箱再次被选中的次数

如果没有这层观测，后面很难判断“是不是确实有必要继续做远端清理”。

## 暂时不要做

以下事项当前阶段不建议直接上。

### 1. 不要把远端删除塞进注册主链路

不建议：

- 创建邮箱失败后立刻同步删远端
- OTP 超时后立刻删远端
- 任务失败收尾时顺手删远端

这会明显增加：

- 网络依赖
- 任务耗时
- 删除失败带来的连锁异常
- 对延后任务的误伤概率

### 2. 不要把“所有失败邮箱”一刀切删掉

很多失败是流程问题，不是邮箱本身问题。

如果简单粗暴地删，会造成：

- 本可重试的任务被提前破坏
- 域名或代理波动被误判成邮箱失效
- 队列成功率下降

### 3. 不要在 `cloud_mail` 现状下假定已经支持远端真实删除

当前 `cloud_mail` 侧的 `delete_email()` 只是删本地缓存，不是真删远端。

在没有补齐远端删除能力前，不应把它当作真正的双删方案。

## 是否有必要同步删除 Cloudflare 远端数据

结论是：有价值，但优先级低于“本地治理和选择层屏蔽”。

只有在下面情况明显出现时，才建议认真推进远端物理删除：

- Cloudflare 侧邮箱数据增长很快，已经影响存储成本
- 远端邮箱列表查询明显变慢
- 远端接口性能因历史垃圾数据下降
- 有安全或合规要求，需要尽快清除无效邮箱

如果暂时没有这些明显问题，那么优先把系统做成：

- 坏邮箱不再参与新任务
- 队列吞吐不下降
- 延后重试不被误删打断

这比立即推进双删更有实际收益。

## 分服务建议

### Freemail

`freemail` 已具备真实远端删除接口，因此它更适合先做异步远端清理试点。

建议：

- 先接入生命周期状态管理
- 先跑隔离观察
- 稳定后再开启低并发异步真实删除

### Cloud Mail

`cloud_mail` 当前不适合直接推进“同步双删”。

建议：

- 先做本地治理与状态隔离
- 后续单独在 `cloud-mail` 服务端补真实删除接口或后台清理任务
- 完成真实删除闭环后，再由 `cc` 通过清理器调用

## 推荐执行顺序

建议按下面顺序推进。

1. 先做邮箱状态治理，阻止坏邮箱再次被选中。
2. 再做失败原因分类，不把暂时失败误判成永久无效。
3. 再做邮箱生命周期表。
4. 再做异步清理器，但先只隔离，不做真实删除。
5. 观察稳定后，对 `freemail` 开启真实远端删除。
6. 最后再考虑给 `cloud_mail` 补齐真实远端删除闭环。

## 最小实施清单

如果只做最小必要改动，建议先落这几项：

- 增加邮箱状态字段或单独治理表
- 失败时只打标签，不删远端
- 新任务选择邮箱时跳过 `quarantined` 和 `invalid_hard`
- 对 `deferred` 任务设置邮箱保护
- 增加后台清理器，但初期只标记 `purge_ready`
- 增加基础统计和观测指标

这套最小方案的价值是：

- 能明显减少坏邮箱复用
- 对现有功能侵入较小
- 几乎不会影响队列吞吐
- 后续还能平滑升级到真实远端清理

## 最终建议

当前最合理的方向不是“立刻同步删除 Cloudflare 远端无效邮箱”，而是：

- 先把无效邮箱治理做好
- 先让坏邮箱退出选择层
- 先保证队列速度和重试稳定性
- 再用异步、低优先级、可观察的方式推进远端清理

如果后续仍然出现远端数据膨胀、成本上升或接口变慢，再继续补 `freemail` / `cloud_mail` 的真实删除闭环，会更稳，也更符合当前项目的收益顺序。
