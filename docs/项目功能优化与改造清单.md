# 项目功能优化与改造清单

本文档仅聚焦功能、稳定性、工程实现与可维护性，不讨论样式层问题。

## 一、当前值得优化的地方

### 1. 注册与批量任务状态存在多套事实来源

当前任务状态同时分散在内存结构和数据库中维护，批量任务还在多个协程里直接修改共享字典。短期内虽然可运行，但天然容易出现以下问题：

- 页面展示状态与数据库最终状态不一致
- WebSocket 推送状态与接口查询结果不一致
- 取消、重启恢复、批量并发完成等边界场景容易出现状态漂移
- 后续修复时需要同时修改多处状态写入逻辑，容易漏改

这类问题已经在前一轮一致性修复中实际暴露过，说明它不是理论风险，而是已经进入真实维护成本区间。

### 2. 核心热点模块过大，功能演进风险持续升高

当前多个核心文件已经发展为巨石模块，职责混杂，路由、编排、状态机、外部调用和兜底逻辑耦合在一起。典型文件包括：

- `src/core/register.py`
- `src/web/routes/registration.py`
- `src/web/routes/payment.py`
- `src/web/routes/accounts.py`
- `src/web/routes/auto_team.py`

这会直接带来几个功能层问题：

- 修改一个分支逻辑时容易误伤其他链路
- 同类逻辑在不同位置重复实现，逐渐形成行为漂移
- 新增测试难以下手，回归验证成本越来越高
- 边界条件只能不断叠加补丁，状态机越来越难以推理

### 3. 数据库迁移仍以启动时手写修补为主，长期风险高

仓库里虽然有 Alembic，但实际结构修复主要依赖启动时执行的手写迁移逻辑。这种方式上线快，但长期会带来明显问题：

- 历史库结构可能在不同机器上逐步分叉
- 部分迁移失败时只打 warning，可能留下半迁移状态
- 迁移逻辑与运行逻辑混在一起，启动行为不可预测
- 后续排查问题时难以确认某个库到底处于哪一版 schema

### 4. 配置来源不单一，实际生效配置不透明

配置文件说明写的是“完全基于数据库存储”，但实际仍然存在环境变量覆盖数据库配置的行为。当前混合模式的主要问题是：

- UI 里看到的值不一定是真正生效的值
- 不同部署方式下行为可能不一致
- 排查线上问题时需要同时检查数据库、环境变量和启动参数
- 容易出现“配置看起来没问题，但服务实际跑的是另一套值”

### 5. 邮箱服务存在新旧实现并存，后续功能维护易漂移

仓库里保留了多组新旧并存实现，例如：

- `cloud_mail.py` 与 `cloudmail.py`
- `temp_mail.py` 与 `tempmail.py`
- `outlook/service.py` 与 `outlook_legacy_mail.py`

虽然当前工厂注册大体已指向主实现，但旧代码仍在仓库中，存在以下问题：

- 后续维护时容易误改旧实现
- 新功能只接入一条链路，另一条旧链路 silently 漂移
- 测试不一定覆盖到真实运行路径
- 阅读成本高，真实入口不够清晰

### 6. 异常处理过宽，诊断质量与功能可观测性不足

当前代码中存在大量宽泛的 `except Exception`，并伴随多处 `pass` 或仅 warning 后继续执行的处理方式。这会带来：

- 真正失败原因被吞掉，最终只剩“未知错误”或状态异常
- 外部依赖错误与本地状态写入错误无法清晰区分
- 某些功能已经失败，但链路仍继续往后执行，产生二次污染
- 测试即使覆盖到异常分支，也不容易验证真实根因

### 7. 邮箱治理注册表目前仍是本地 JSON 旁路实现

`MailboxRegistry` 现在是一个轻量旁路设计，优点是改动小、风险可控；但如果继续承载更多治理能力，就会暴露限制：

- 仅有进程内线程锁，没有跨进程协调
- 多实例部署下不能保证一致性
- 读写是全量 JSON 文件，不适合持续扩展
- 不利于查询、审计、统计和历史分析

因此它适合作为过渡方案，不适合作为长期治理底座。

### 8. CRUD 层事务边界偏碎，不利于复杂功能保持原子性

部分底层 CRUD 方法内部直接 `commit`。这种设计在简单场景下方便，但对于复杂业务链路有明显缺点：

- 上层无法把多步业务放进同一个事务
- 中间一步失败时容易留下半成功数据
- 审计、任务状态、业务实体很难做到一致提交
- 后续增加补偿逻辑时复杂度明显变高

### 9. 高风险模块测试覆盖仍偏薄

现有测试对注册链路的覆盖明显比以前更好，但对于以下模块，覆盖深度还不够：

- `payment`
- `accounts`
- `auto_team`
- `selfcheck`
- `websocket`

当前的主要问题不是“完全没测试”，而是高风险场景缺少真正能挡回归的测试，例如：

- 并发状态更新
- 取消竞争
- 启动恢复
- WebSocket 状态广播
- 复杂任务的部分成功 / 部分失败收尾

### 10. 时间写入口径未完全统一

项目里大部分数据库写入使用 UTC，但部分元数据仍使用本地时间写入。继续混用会导致：

- 排查时序问题时理解成本上升
- 多时区部署后日志与数据库时间不一致
- 治理、统计、排序、恢复判断可能出现偏差

这类问题短期不一定爆炸，但属于会持续积累的基础一致性债务。

## 二、改造清单

下面按照优先级给出建议的改造顺序。

## P0：应优先启动的改造

### 1. 收敛任务状态为单一真源

目标：

- 单任务状态、批量任务状态、取消状态、完成状态统一由一套状态仓库维护
- 接口查询、WebSocket 推送、重启恢复都基于同一事实来源

建议改造：

- 将注册任务和批量任务状态统一收敛到数据库层
- 内存层仅作为短期缓存或广播辅助，不再承担最终状态职责
- 封装统一的任务状态服务，禁止路由层和协程直接散写状态
- 对取消、完成、失败、deferred 建立统一收口函数

验收标准：

- 任一任务的接口查询结果、数据库状态、WebSocket 最终广播保持一致
- 重启恢复后不会出现“库里已结束但内存仍运行”的漂移

### 2. 拆分注册链路中的路由、编排与状态机

目标：

- 将注册流程从巨石函数拆成可维护的功能模块

建议改造：

- `registration route` 只负责参数校验、鉴权和响应封装
- 新增注册应用服务层，负责任务创建、调度、状态写入
- 将批量执行器从路由文件拆出，独立为 batch orchestrator
- 将取消、defer、失败诊断、成功收尾统一抽为可复用组件

验收标准：

- `src/web/routes/registration.py` 明显降体积
- 批量并发与单任务流程共享同一套状态收口逻辑

### 3. 补齐状态一致性与并发边界测试

目标：

- 让当前最容易回归的地方有明确保护网

建议改造：

- 增加单任务取消竞争测试
- 增加批量任务取消与并发完成冲突测试
- 增加重启恢复测试
- 增加 WebSocket 状态广播顺序测试
- 增加 deferred 重试次数耗尽后的状态落库测试

验收标准：

- 状态一致性问题能通过测试复现并被锁住
- 后续改动不需要每次都靠手工隔离验证兜底

## P1：应在 P0 后尽快推进的改造

### 4. 将数据库迁移回归 Alembic 主路径

目标：

- 让 schema 变更可追踪、可回滚、可审计

建议改造：

- 把 `migrate_tables()` 中的结构迁移逐步迁回 Alembic revision
- 启动阶段只做版本检查和必要提示，不再隐式大量改表
- 明确 SQLite 与 PostgreSQL 支持范围和迁移策略

验收标准：

- 新环境初始化与旧环境升级走同一条标准迁移路径
- 可以明确知道数据库当前 schema 版本

### 5. 统一配置优先级与配置可观测性

目标：

- 明确“到底哪个配置生效”

建议改造：

- 将配置分成“启动层配置”和“业务层配置”
- 只允许少量启动层配置被环境变量覆盖
- 在设置接口或诊断接口中返回配置来源
- 修正文档中“完全基于数据库存储”这类已不完全准确的表述

验收标准：

- 排查配置类问题时不需要猜测值来自哪里
- UI 展示值与实际运行值可核对

### 6. 收口邮箱服务实现，明确主路径和废弃路径

目标：

- 减少服务实现漂移和维护歧义

建议改造：

- 明确保留的邮箱服务实现
- 对旧实现加 `deprecated` 标记或迁移出主路径
- 工厂注册只保留唯一主实现映射
- 增加每类邮箱服务的契约测试

验收标准：

- 同一服务类型只有一条明确的真实执行路径
- 新功能接入时无需猜测该改哪份实现

### 7. 收紧异常分类与诊断输出

目标：

- 提高故障诊断精度，降低“未知错误”比例

建议改造：

- 区分外部依赖失败、参数错误、状态写入失败、迁移失败、广播失败
- 核心链路禁止直接 `except Exception: pass`
- 对关键异常打稳定 reason_code 或 error_category
- 将诊断信息沉淀到任务记录和日志结构中

验收标准：

- 常见失败场景能定位到明确类别
- 日志可直接区分是上游失败还是本地一致性失败

## P2：中期治理与结构优化

### 8. 将邮箱治理从 JSON 文件迁移到数据库表

目标：

- 为后续邮箱子域治理、失败统计、隔离与恢复提供稳定底座

建议改造：

- 为邮箱治理建立正式表结构
- 保留当前 JSON 作为一次性迁移来源
- 增加按邮箱、子域、服务类型的查询与统计能力
- 为未来多实例部署预留一致性基础

验收标准：

- 治理数据可查询、可审计、可统计
- 不再依赖本地 JSON 文件作为长期状态存储

### 9. 调整 CRUD 与事务边界

目标：

- 支撑复杂功能的原子提交与可补偿设计

建议改造：

- 基础 CRUD 减少内部 `commit`
- 将事务控制上移到 service / use case 层
- 对“建任务 + 写状态 + 写审计”这类多步操作使用统一事务

验收标准：

- 复杂链路可以做到同事务提交或清晰补偿
- 半成功脏数据明显减少

### 10. 统一全项目时间写入策略

目标：

- 消除时区混写导致的诊断偏差

建议改造：

- 持久化层统一使用 UTC
- 元数据、日志、任务状态时间统一口径
- 展示层按用户时区转换，不在底层混写本地时间

验收标准：

- 任意时间字段的解释口径一致
- 任务时序分析不再依赖人工猜测时区

## 三、建议执行顺序

建议按以下顺序推进：

1. 任务状态单一真源
2. 注册链路与批量编排拆分
3. 一致性与并发边界测试补齐
4. 数据库迁移回归 Alembic
5. 配置优先级统一
6. 邮箱服务实现收口
7. 异常分类与诊断结构化
8. 邮箱治理落库
9. CRUD 事务边界调整
10. 时间口径统一

## 四、整体评价

当前项目已经具备可用的业务能力，也有一定测试基础，说明主体功能并不是“不可维护”，但随着注册、支付、账户、团队等能力不断叠加，核心问题已经从“有没有功能”转为“功能能否持续稳定演进”。

从这次巡检看，项目最主要的债务不是单点 bug，而是以下三类结构性问题已经开始互相放大：

- 状态来源分散
- 巨石模块过大
- 迁移、配置、诊断等基础设施口径不统一

结论是：项目当前处于“还能继续叠功能，但再不收口结构，后续每次改动成本都会继续抬高”的阶段。

评价：

- 短期可用性：较好
- 中期演进性：一般
- 一致性基础：偏弱
- 后续改造价值：高

如果只允许优先做最值的改造，那么最应该先做的不是继续堆新功能，而是先把任务状态真源、注册链路拆分和关键一致性测试这三项落下来。
