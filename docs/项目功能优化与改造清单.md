# 项目功能优化与改造清单

本文档仅聚焦功能、稳定性、工程实现与可维护性，不讨论样式层问题。

## 一、当前值得优化的地方

### 1. 注册与批量任务状态存在多套事实来源

当前任务状态同时分散在内存结构和数据库中维护，批量任务还在多个协程里直接修改共享字典。短期内虽然可运行，但天然容易出现以下问题：

- 页面展示状态与数据库最终状态不一致
- WebSocket 推送状态与接口查询结果不一致
- 取消、重启恢复、批量并发完成等边界场景容易出现状态漂移
- 后续修复时需要同时修改多处状态写入逻辑，容易漏改

这类问题已经在前一轮一致性修复中实际暴露过，说明它不是理论风险，而是已经进入真实维护成本区间。

### 2. 核心热点模块过大，功能演进风险持续升高

当前多个核心文件已经发展为巨石模块，职责混杂，路由、编排、状态机、外部调用和兜底逻辑耦合在一起。典型文件包括：

- `src/core/register.py`
- `src/web/routes/registration.py`
- `src/web/routes/payment.py`
- `src/web/routes/accounts.py`
- `src/web/routes/auto_team.py`

这会直接带来几个功能层问题：

- 修改一个分支逻辑时容易误伤其他链路
- 同类逻辑在不同位置重复实现，逐渐形成行为漂移
- 新增测试难以下手，回归验证成本越来越高
- 边界条件只能不断叠加补丁，状态机越来越难以推理

### 3. 数据库迁移仍以启动时手写修补为主，长期风险高

仓库里虽然有 Alembic，但实际结构修复主要依赖启动时执行的手写迁移逻辑。这种方式上线快，但长期会带来明显问题：

- 历史库结构可能在不同机器上逐步分叉
- 部分迁移失败时只打 warning，可能留下半迁移状态
- 迁移逻辑与运行逻辑混在一起，启动行为不可预测
- 后续排查问题时难以确认某个库到底处于哪一版 schema

### 4. 配置来源不单一，实际生效配置不透明

配置文件说明写的是“完全基于数据库存储”，但实际仍然存在环境变量覆盖数据库配置的行为。当前混合模式的主要问题是：

- UI 里看到的值不一定是真正生效的值
- 不同部署方式下行为可能不一致
- 排查线上问题时需要同时检查数据库、环境变量和启动参数
- 容易出现“配置看起来没问题，但服务实际跑的是另一套值”

### 5. 邮箱服务存在新旧实现并存，后续功能维护易漂移

仓库里保留了多组新旧并存实现，例如：

- `cloud_mail.py` 与 `cloudmail.py`
- `temp_mail.py` 与 `tempmail.py`
- `outlook/service.py` 与 `outlook_legacy_mail.py`

虽然当前工厂注册大体已指向主实现，但旧代码仍在仓库中，存在以下问题：

- 后续维护时容易误改旧实现
- 新功能只接入一条链路，另一条旧链路 silently 漂移
- 测试不一定覆盖到真实运行路径
- 阅读成本高，真实入口不够清晰

### 6. 异常处理过宽，诊断质量与功能可观测性不足

当前代码中存在大量宽泛的 `except Exception`，并伴随多处 `pass` 或仅 warning 后继续执行的处理方式。这会带来：

- 真正失败原因被吞掉，最终只剩“未知错误”或状态异常
- 外部依赖错误与本地状态写入错误无法清晰区分
- 某些功能已经失败，但链路仍继续往后执行，产生二次污染
- 测试即使覆盖到异常分支，也不容易验证真实根因

### 7. 邮箱治理注册表目前仍是本地 JSON 旁路实现

`MailboxRegistry` 现在是一个轻量旁路设计，优点是改动小、风险可控；但如果继续承载更多治理能力，就会暴露限制：

- 仅有进程内线程锁，没有跨进程协调
- 多实例部署下不能保证一致性
- 读写是全量 JSON 文件，不适合持续扩展
- 不利于查询、审计、统计和历史分析

因此它适合作为过渡方案，不适合作为长期治理底座。

### 8. CRUD 层事务边界偏碎，不利于复杂功能保持原子性

部分底层 CRUD 方法内部直接 `commit`。这种设计在简单场景下方便，但对于复杂业务链路有明显缺点：

- 上层无法把多步业务放进同一个事务
- 中间一步失败时容易留下半成功数据
- 审计、任务状态、业务实体很难做到一致提交
- 后续增加补偿逻辑时复杂度明显变高

### 9. 高风险模块测试覆盖仍偏薄

现有测试对注册链路的覆盖明显比以前更好，但对于以下模块，覆盖深度还不够：

- `payment`
- `accounts`
- `auto_team`
- `selfcheck`
- `websocket`

当前的主要问题不是“完全没测试”，而是高风险场景缺少真正能挡回归的测试，例如：

- 并发状态更新
- 取消竞争
- 启动恢复
- WebSocket 状态广播
- 复杂任务的部分成功 / 部分失败收尾

### 10. 时间写入口径未完全统一

项目里大部分数据库写入使用 UTC，但部分元数据仍使用本地时间写入。继续混用会导致：

- 排查时序问题时理解成本上升
- 多时区部署后日志与数据库时间不一致
- 治理、统计、排序、恢复判断可能出现偏差

这类问题短期不一定爆炸，但属于会持续积累的基础一致性债务。

## 二、改造清单

下面按照优先级给出建议的改造顺序。

## P0：应优先启动的改造

### 1. 收敛任务状态为单一真源

目标：

- 单任务状态、批量任务状态、取消状态、完成状态统一由一套状态仓库维护
- 接口查询、WebSocket 推送、重启恢复都基于同一事实来源

建议改造：

- 将注册任务和批量任务状态统一收敛到数据库层
- 内存层仅作为短期缓存或广播辅助，不再承担最终状态职责
- 封装统一的任务状态服务，禁止路由层和协程直接散写状态
- 对取消、完成、失败、deferred 建立统一收口函数

验收标准：

- 任一任务的接口查询结果、数据库状态、WebSocket 最终广播保持一致
- 重启恢复后不会出现“库里已结束但内存仍运行”的漂移

### 2. 拆分注册链路中的路由、编排与状态机

目标：

- 将注册流程从巨石函数拆成可维护的功能模块

建议改造：

- `registration route` 只负责参数校验、鉴权和响应封装
- 新增注册应用服务层，负责任务创建、调度、状态写入
- 将批量执行器从路由文件拆出，独立为 batch orchestrator
- 将取消、defer、失败诊断、成功收尾统一抽为可复用组件

验收标准：

- `src/web/routes/registration.py` 明显降体积
- 批量并发与单任务流程共享同一套状态收口逻辑

### 3. 补齐状态一致性与并发边界测试

目标：

- 让当前最容易回归的地方有明确保护网

建议改造：

- 增加单任务取消竞争测试
- 增加批量任务取消与并发完成冲突测试
- 增加重启恢复测试
- 增加 WebSocket 状态广播顺序测试
- 增加 deferred 重试次数耗尽后的状态落库测试

验收标准：

- 状态一致性问题能通过测试复现并被锁住
- 后续改动不需要每次都靠手工隔离验证兜底

## P1：应在 P0 后尽快推进的改造

### 4. 将数据库迁移回归 Alembic 主路径

目标：

- 让 schema 变更可追踪、可回滚、可审计

建议改造：

- 把 `migrate_tables()` 中的结构迁移逐步迁回 Alembic revision
- 启动阶段只做版本检查和必要提示，不再隐式大量改表
- 明确 SQLite 与 PostgreSQL 支持范围和迁移策略

验收标准：

- 新环境初始化与旧环境升级走同一条标准迁移路径
- 可以明确知道数据库当前 schema 版本

### 5. 统一配置优先级与配置可观测性

目标：

- 明确“到底哪个配置生效”

建议改造：

- 将配置分成“启动层配置”和“业务层配置”
- 只允许少量启动层配置被环境变量覆盖
- 在设置接口或诊断接口中返回配置来源
- 修正文档中“完全基于数据库存储”这类已不完全准确的表述

验收标准：

- 排查配置类问题时不需要猜测值来自哪里
- UI 展示值与实际运行值可核对

### 6. 收口邮箱服务实现，明确主路径和废弃路径

目标：

- 减少服务实现漂移和维护歧义

建议改造：

- 明确保留的邮箱服务实现
- 对旧实现加 `deprecated` 标记或迁移出主路径
- 工厂注册只保留唯一主实现映射
- 增加每类邮箱服务的契约测试

验收标准：

- 同一服务类型只有一条明确的真实执行路径
- 新功能接入时无需猜测该改哪份实现

### 7. 收紧异常分类与诊断输出

目标：

- 提高故障诊断精度，降低“未知错误”比例

建议改造：

- 区分外部依赖失败、参数错误、状态写入失败、迁移失败、广播失败
- 核心链路禁止直接 `except Exception: pass`
- 对关键异常打稳定 reason_code 或 error_category
- 将诊断信息沉淀到任务记录和日志结构中

验收标准：

- 常见失败场景能定位到明确类别
- 日志可直接区分是上游失败还是本地一致性失败

## P2：中期治理与结构优化

### 8. 将邮箱治理从 JSON 文件迁移到数据库表

目标：

- 为后续邮箱子域治理、失败统计、隔离与恢复提供稳定底座

建议改造：

- 为邮箱治理建立正式表结构
- 保留当前 JSON 作为一次性迁移来源
- 增加按邮箱、子域、服务类型的查询与统计能力
- 为未来多实例部署预留一致性基础

验收标准：

- 治理数据可查询、可审计、可统计
- 不再依赖本地 JSON 文件作为长期状态存储

### 9. 调整 CRUD 与事务边界

目标：

- 支撑复杂功能的原子提交与可补偿设计

建议改造：

- 基础 CRUD 减少内部 `commit`
- 将事务控制上移到 service / use case 层
- 对“建任务 + 写状态 + 写审计”这类多步操作使用统一事务

验收标准：

- 复杂链路可以做到同事务提交或清晰补偿
- 半成功脏数据明显减少

### 10. 统一全项目时间写入策略

目标：

- 消除时区混写导致的诊断偏差

建议改造：

- 持久化层统一使用 UTC
- 元数据、日志、任务状态时间统一口径
- 展示层按用户时区转换，不在底层混写本地时间

验收标准：

- 任意时间字段的解释口径一致
- 任务时序分析不再依赖人工猜测时区

## 三、建议执行顺序

建议按以下顺序推进：

1. 任务状态单一真源
2. 注册链路与批量编排拆分
3. 一致性与并发边界测试补齐
4. 数据库迁移回归 Alembic
5. 配置优先级统一
6. 邮箱服务实现收口
7. 异常分类与诊断结构化
8. 邮箱治理落库
9. CRUD 事务边界调整
10. 时间口径统一

## 四、进一步拆分后的实施计划

本节将改造清单继续拆成可以直接排期的实施阶段。建议每个阶段都单独提交、单独回归、单独验收，避免再次形成“大补丁一次性落地”的风险。

### 阶段 A：任务状态真源收敛

目标：

- 先解决最核心的一致性问题
- 为后续注册链路拆分提供稳定底座

子任务：

- 梳理单任务、批量任务、取消状态、deferred 状态的所有写入口
- 明确数据库字段与运行态字段的最终语义
- 新建统一任务状态服务，封装状态变更 API
- 替换路由层和协程中分散的直接写状态逻辑
- 为 WebSocket 广播改成订阅统一状态变更结果，而不是自行拼状态

建议交付物：

- 统一任务状态服务模块
- 状态变更时序图
- 任务状态字段说明文档

阶段风险：

- 改动点分布广，容易漏掉旧入口
- 如果一次性替换过多，可能引入新的状态回归

控制方式：

- 先引入统一服务，再逐步迁移旧调用点
- 保留短期兼容层，先收口再删旧逻辑

阶段验收：

- 单任务 API 查询、DB 状态、WS 广播结果一致
- 批量任务在取消、失败、完成时结果一致

### 阶段 B：注册链路应用层拆分

目标：

- 把注册能力从巨石路由和巨石引擎中拆出清晰边界

子任务：

- 拆出注册任务创建服务
- 拆出注册任务执行服务
- 拆出批量执行编排器
- 拆出统一的取消收口逻辑
- 拆出统一的结果归档与诊断收口逻辑

建议交付物：

- `registration application service`
- `batch orchestrator`
- `result finalizer`
- `cancellation coordinator`

阶段风险：

- 旧逻辑里有大量隐式耦合状态，拆分时可能漏带上下文

控制方式：

- 先按职责抽函数，再抽模块，再抽服务
- 每抽一层都先补针对性回归测试

阶段验收：

- `registration.py` 体积明显下降
- 单任务与批量任务共享同一套结果收口逻辑
- 不再出现不同入口下取消原因不一致

### 阶段 C：一致性与并发测试加固

目标：

- 把最容易回归的行为变成自动化保护

子任务：

- 增加任务取消竞争测试
- 增加批量并发完成测试
- 增加 deferred 重试上限测试
- 增加重启恢复测试
- 增加 WebSocket 状态广播与日志顺序测试

建议交付物：

- `tests/test_registration_task_consistency.py`
- `tests/test_registration_batch_recovery.py`
- `tests/test_registration_websocket_state.py`

阶段风险：

- 现有代码耦合较重，测试搭建成本较高

控制方式：

- 先补 service 层测试，再补 route 层测试
- 优先 mock 外部依赖，只测状态机和持久化结果

阶段验收：

- 核心一致性问题可由自动化测试复现并锁住
- 不再依赖人工近线上验证作为唯一兜底

### 阶段 D：数据库迁移治理

目标：

- 把结构变更从“启动时修库”转成“显式迁移”

子任务：

- 梳理现有 `migrate_tables()` 中所有结构性迁移动作
- 区分一次性历史修复和长期 schema 变更
- 按主题补 Alembic revision
- 启动阶段降级为版本检查和提示

建议交付物：

- Alembic 迁移脚本
- schema 版本说明
- 历史库升级说明

阶段风险：

- 历史数据库样本可能不完全一致

控制方式：

- 先收集几份典型旧库做演练
- 为高风险迁移加备份和可回滚说明

阶段验收：

- 新环境与旧环境都能走标准迁移路径
- 不再依赖启动时大量隐式 `ALTER TABLE`

### 阶段 E：配置与运行时可观测性收口

目标：

- 让“当前到底生效了什么配置”可以直接确认

子任务：

- 梳理数据库配置、环境变量配置、启动参数的优先级
- 分类定义启动层配置与业务层配置
- 为设置页或诊断接口增加配置来源展示
- 修正文档与代码注释中的错误描述

建议交付物：

- 配置优先级说明
- 生效配置诊断接口
- 配置来源字段展示

阶段风险：

- 变更优先级后，部分现有部署方式行为可能变化

控制方式：

- 先保持兼容，再发出弃用提示
- 为高风险配置增加启动日志告警

阶段验收：

- 能明确看到每个关键配置的最终值和来源
- 排查配置问题时不需要人工逐项猜测

### 阶段 F：邮箱服务路径收口

目标：

- 保证同一服务类型只有一条真实执行路径

子任务：

- 梳理所有邮箱服务实现及当前引用关系
- 标记主实现、兼容实现、待淘汰实现
- 收敛工厂映射
- 增加契约测试，保证不同服务满足统一接口行为

建议交付物：

- 邮箱服务矩阵
- 主路径说明
- 契约测试集

阶段风险：

- 清理旧实现时可能影响历史配置兼容

控制方式：

- 先加 deprecated 标记，再迁移引用，再删除旧实现

阶段验收：

- 新增或修改邮箱服务时，开发入口明确
- 不再存在“看起来能用但实际不走”的旧实现歧义

### 阶段 G：异常分类与诊断结构化

目标：

- 让失败原因更精确、更可统计、更可回放

子任务：

- 梳理高频异常类型
- 定义统一错误分类字段
- 对注册、支付、账号、团队等主链路收紧异常处理
- 禁止关键路径静默吞错

建议交付物：

- 错误分类规范
- reason_code / error_category 对照表
- 结构化诊断字段说明

阶段风险：

- 收紧异常后，短期内可能暴露更多历史隐患

控制方式：

## 六、近期最值得先落地的三项

如果只允许先做三项，我建议顺序固定为：

## 七、第一阶段落地作战表

本节只针对最值得优先落地的三项，继续细化到可直接开工的程度。

### 7.1 第一阶段目标

第一阶段只做三件事：

1. 收敛任务状态真源
2. 拆分注册链路关键职责
3. 补齐状态一致性测试

控制原则：

- 不追求一步到位大重构
- 每一步都要可独立提交、独立回归、独立回滚
- 先测试兜底，再收口逻辑，再删旧实现

### 7.2 建议的开发顺序

建议按以下顺序推进：

1. 先补一致性测试骨架
2. 再引入统一任务状态服务
3. 然后替换单任务状态写入口
4. 再替换批量任务状态写入口
5. 再拆注册应用服务与批量编排器
6. 最后删除已失效的旧状态写法

这样做的原因是：

- 没有测试先收口状态，回归风险过大
- 没有统一状态服务先拆模块，容易把旧问题带进新结构
- 批量任务的状态边界比单任务更复杂，适合放在单任务之后处理

- 线上问题排查会从“猜状态是哪里写坏了”变成“直接看统一状态收口点”

## 八、整体评价

当前项目已经具备可用的业务能力，也有一定测试基础，说明主体功能并不是“不可维护”，但随着注册、支付、账户、团队等能力不断叠加，核心问题已经从“有没有功能”转为“功能能否持续稳定演进”。

从这次巡检看，项目最主要的债务不是单点 bug，而是以下三类结构性问题已经开始互相放大：

- 状态来源分散
- 巨石模块过大
- 迁移、配置、诊断等基础设施口径不统一

结论是：项目当前处于“还能继续叠功能，但再不收口结构，后续每次改动成本都会继续抬高”的阶段。

评价：

- 短期可用性：较好
- 中期演进性：一般
- 一致性基础：偏弱
- 后续改造价值：高

如果只允许优先做最值的改造，那么最应该先做的不是继续堆新功能，而是先把任务状态真源、注册链路拆分和关键一致性测试这三项落下来。

### Issue 1：补齐注册任务状态一致性测试

背景：

当前注册任务状态在内存、数据库和 WebSocket 三处存在并行维护，历史上已经出现过最终状态与原因不一致的问题。后续要推进状态真源收敛，必须先补齐测试保护网。

目标：

- 为单任务取消、失败、deferred、最终状态广播建立基础回归测试

- 新增单任务状态一致性测试
- 新增 WebSocket 最终状态一致性测试
- 不在本任务中改造主业务逻辑

建议修改：

- 新增 `tests/test_registration_task_consistency.py`
- 新增 `tests/test_registration_websocket_state.py`
